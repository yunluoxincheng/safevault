# Design: Token Refresh Improvement

## Context

SafeVault Android 应用使用 JWT token 进行认证，当前刷新机制存在多个问题导致用户体验不佳。后端 token 配置为：
- Access Token: 24小时过期
- Refresh Token: 30天过期

## Goals / Non-Goals

**Goals:**
- 修复错误传播，让 UI 能够感知 token 过期
- 添加刷新失败后的用户通知
- 实现主动刷新，减少用户遇到 401 的情况
- 改进并发刷新机制，提高稳定性

**Non-Goals:**
- 不修改后端 token 过期时间配置
- 不改变 JWT token 的签发和验证逻辑
- 不引入第三方认证库

## Decisions

### 决策 1：错误传播方式

**选择：** 使用自定义异常 + RxJava onError 链

**原因：**
- 保持与现有 RxJava 架构一致
- 可以区分不同类型的错误（认证错误、网络错误、业务错误）
- UI 层可以根据异常类型做不同处理

**替代方案：**
- 使用 LiveData/ErrorState 对象 - 会增加复杂度
- 直接在拦截器中跳转登录 - 违反单一职责原则

### 决策 2：刷新失败通知机制

**选择：** 使用本地广播（LocalBroadcastManager 或普通广播）

**原因：**
- 广播机制是 Android 标准，解耦性好
- 可以在 BaseActivity 中统一处理
- 性能开销小

**替代方案：**
- 使用 LiveData/RxJava Bus - 需要额外的依赖注入
- 使用 EventBus - 引入第三方库

### 决策 3：主动刷新时机

**选择：** 应用启动时检查，剩余时间 < 5分钟时刷新

**原因：**
- 5分钟足够处理网络延迟和刷新失败
- 启动时检查不会影响用户操作
- 避免过于频繁的刷新请求

**替代方案：**
- 每次 API 调用前检查 - 性能开销大
- 使用后台定时任务 - 增加复杂度和电量消耗

### 决策 4：并发刷新机制

**选择：** 使用 AtomicReference + CountDownLatch + 请求队列

**原因：**
- 保持现有同步刷新架构
- 使用队列确保请求按顺序处理
- AtomicReference 提供线程安全的状态管理

**替代方案：**
- 使用 RxJava 的 retryWhen - 只能重试，不能批量处理等待的请求
- 使用 RxJava 的 defer - 逻辑复杂，难调试

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 主动刷新可能失败 | 用户仍会遇到 401 | 保留被动刷新作为后备 |
| 广播接收器未注册 | 刷新失败后无响应 | 在 BaseActivity 中统一注册 |
| JWT 解析可能出错 | 主动刷新失效 | 添加异常捕获，降级到被动刷新 |
| 并发场景复杂 | 难以测试 | 编写单元测试覆盖各种场景 |

## Migration Plan

### 步骤
1. 先实现方案 A 和 B（高优先级）
2. 测试验证错误传播和通知机制
3. 再实现方案 C（主动刷新）
4. 最后实现方案 D（并发优化）
5. 完整的集成测试

### 回滚计划
- 每个方案独立实现，可以单独回滚
- 保留原有代码路径，使用特性开关控制

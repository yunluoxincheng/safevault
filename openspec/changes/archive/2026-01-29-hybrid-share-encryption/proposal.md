# 提案：联系人分享混合加密方案

## 变更 ID
`hybrid-share-encryption`

## 问题陈述

当前联系人分享功能使用纯 RSA 加密，存在以下问题：

1. **数据大小限制**：RSA-2048 位密钥使用 OAEPWithSHA-256AndMGF1Padding 最多只能加密 214 字节
2. **分享数据包过大**：`ShareDataPacket` 序列化后约 720 字节，包含：
   - 发送者公钥（Base64，~400 字节）
   - 密码详细信息（~200 字节）
   - 权限、时间戳等（~100 字节）
3. **加密失败**：用户尝试分享密码时提示 "Data too large for RSA encryption: 720 > -6"
4. **计算错误**：`getMaxBlockSize()` 方法使用错误的计算方式（`publicKey.getEncoded().length`），导致返回负数

## 影响范围

- **用户**：无法通过联系人分享功能分享密码
- **功能**：联系人分享完全不可用
- **安全**：需要确保新方案不降低安全性

## 解决方案概述

采用**混合加密方案（RSA + AES）**：
1. 用接收方 RSA 公钥加密随机生成的 AES-256 密钥
2. 用 AES-256-GCM 加密实际的分享数据
3. 保留数字签名验证发送方身份

### 优势

- ✅ **无数据大小限制**：AES 可加密任意大小数据
- ✅ **安全性最高**：结合 RSA 密钥交换和 AES 数据加密
- ✅ **性能更好**：RSA 只加密 32 字节，AES 处理大数据快
- ✅ **行业标准**：PGP、S/MIME 等成熟方案的做法
- ✅ **可扩展**：未来支持文件等大数据分享

### 劣势

- ❌ **实现复杂度较高**：需要管理 AES 密钥、IV、加密包格式
- ❌ **协议不兼容**：版本从 1.0 升级到 2.0，需要所有客户端同步升级

## 技术方案

### 数据结构变更

`EncryptedSharePacket` 新增字段：

| 字段 | 类型 | 大小 | 说明 |
|------|------|------|------|
| `version` | String | - | 从 `"1.0"` 升级到 `"2.0"` |
| `encryptedAESKey` | String | ~344字节 | RSA 加密的 32字节 AES 密钥 |
| `iv` | String | 16字节 | AES-GCM 的 12字节 IV |
| `encryptedData` | String | 可变 | AES 加密的分享数据 |
| `signature` | String | ~344字节 | RSA-SHA256 签名 |

### 加密流程

```
发送方                              接收方
─────                              ─────
1. ShareDataPacket (720B)
2. 生成随机 AES-256 密钥 (32B)
3. 用 AES-GCM 加密数据包
4. 用接收方 RSA 公钥加密 AES 密钥
5. 签名原始数据包
───────────────────────────────────>
6. 用接收方 RSA 私钥解密 AES 密钥
7. 用 AES 密钥解密数据
8. 验证签名
9. 得到 ShareDataPacket
```

### 配置决策

- **AES 算法**：AES-256-GCM（业界标准，认证加密）
- **RSA 密钥**：保持 2048 位（现有配置）
- **IV 长度**：12 字节（GCM 标准）
- **协议版本**：2.0（不兼容旧版本）

## 依赖关系

- **前置条件**：无
- **后续影响**：所有使用联系人分享的用户需要升级到新版本

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 实现复杂度高 | 开发周期延长 | 参考 skill 设计，充分测试 |
| 协议不兼容 | 旧版本无法使用 | 强制升级，显示明确提示 |
| AES 密钥管理不当 | 安全风险 | 使用 SecureRandom，每包一密 |
| 签名验证绕过 | 身份伪造 | 确保 verifySignature 正确调用 |

## 替代方案

### 方案二：优化数据结构
将大字段移到明文部分，只加密核心数据。
- 优点：实现较简单
- 缺点：仍有大小限制，未来扩展受限

### 方案三：升级到 RSA-4096
- 优点：实现最简单
- 缺点：性能差，密钥更大，仍有限制

**选择方案一的原因**：长期可维护性和安全性最佳。
